POST 2 — It's Not Just .NET (Week 3)
====================================

"Aspire is only for .NET apps."

I keep hearing this. It's wrong.

I built a demo with:
→ 2 .NET APIs
→ 1 React + TypeScript frontend
→ 1 Python background worker
→ PostgreSQL and Redis

All orchestrated from a single Aspire AppHost.

The AppHost happens to be written in .NET, but it orchestrates ANY language. The Python worker gets its environment variables injected automatically. The React app gets npm install and npm run dev handled for it.

All services appear in the same dashboard. Same logs. Same traces. Same health checks.

3 languages. 1 command. 1 dashboard.

====================================
POST 3 — Developer Onboarding (Week 5)
====================================

A new developer joins your team.

Without Aspire:
1. Read the 40-line README
2. Install Docker, run docker-compose
3. Open Terminal 2, dotnet run the first API
4. Open Terminal 3, dotnet run the second API
5. Open Terminal 4, npm install, npm run dev
6. Open Terminal 5, create a Python venv, pip install, run the worker
7. Something doesn't connect. Check the connection string. It's wrong.
8. Fix it. Restart. Repeat.

With Aspire:
1. dotnet run --project src/OrderCanvas.AppHost
2. Done.

That's not a simplification. That's literally it.

Connection strings? Injected automatically.
Startup order? Handled by WaitFor().
Logs from all services? One dashboard.

I wrote a step-by-step beginner guide showing how this works in practice — with a real multi-language demo app:

https://thecodeman.net/posts/aspire-for-beginners-distributed-applications


====================================
POST 4 — Observability (Week 7)
====================================

Setting up observability for a distributed app used to mean:

→ Install OpenTelemetry packages in every service
→ Configure exporters (Jaeger? Zipkin? OTLP?)
→ Run a collector container
→ Run a UI container (Grafana? Jaeger UI?)
→ Configure Python separately
→ Hope everything sends data to the same place

With Aspire, each .NET service adds one line:

builder.AddServiceDefaults();

That gives you:
• Structured logging
• Distributed tracing
• Metrics
• Health checks

All automatically exported to the built-in Aspire Dashboard.

The Python worker? Aspire injects OTEL_EXPORTER_OTLP_ENDPOINT as an environment variable. No Python-side configuration.

I built a demo called OrderCanvas where you can see a distributed trace flow from React → .NET → Python → PostgreSQL — all in one view.

Full beginner walkthrough here:

https://thecodeman.net/posts/aspire-for-beginners-distributed-applications
